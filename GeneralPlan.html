Что это за проект?
Локальная система управления знаниями: загружаешь PDF, видео, аудио, заметки — система анализирует, строит связи, и потом отвечает на твои вопросы. Всё работает на твоём компьютере, 0 € в месяц, никаких облаков.
Чем отличается от обычного RAG?
Обычный RAG: документ → поиск → ответ.

Твой проект: данные → граф знаний → долговременная память → AI-агент → ответ.

Система не просто находит куски текста — она понимает связи: "Нейросети → Трансформеры → Attention → LLM" и помнит что ты изучал 3 недели назад.
Что пользователь реально получает?
Спрашиваешь "что я читал про трансформеры?" — получаешь структурированный ответ из нескольких источников, связанные материалы, и напоминание что не открывал эту тему 20 дней.

Для портфолио проект демонстрирует: LLM, RAG, векторные БД, графы знаний, backend, frontend, Docker — всё в одном.

Данные проходят через 5 слоёв. Каждый слой — отдельная задача.

📥 Источник — PDF, YouTube, аудио, URL, Markdown
↓
⚙️ Обработка — извлечение текста, очистка, разбивка на чанки
↓
🗄️ Хранение — эмбеддинги в pgvector + граф в NetworkX
↓
🤖 AI-агент — Ollama + Qwen/Gemma + LangChain RAG
↓
💬 Ответ — Next.js чат + граф + история + рекомендации
Эмбеддинги находят семантически похожие тексты (даже если слова разные), а граф хранит явные связи между понятиями. Вместе = намного лучшее качество чем keyword search.

Нажимай на этап чтобы раскрыть задачи.

1Подготовка и инфраструктура 2–3 дня ▶
Создать репозиторий и базовую структуру (backend/, frontend/, docker/)
Почему: CI/CD и README с самого начала экономят время потом
Написать docker-compose.yml: postgres+pgvector, ollama, backend, frontend
Почему: запуск одной командой — это ключевое требование проекта
FastAPI с /health эндпоинтом — убедиться что контейнеры общаются
PostgreSQL + pgvector: установить расширение, создать схему БД
Почему: pgvector нужно ставить отдельно, лучше сразу проверить
Скачать модель через Ollama (qwen3:8b или gemma3:4b)
Почему: модель 4–8 ГБ, скачай заранее
💡 Начни с PostgreSQL + pgvector — это самая частая точка боли. Убедись что векторный поиск работает до того как писать основной код.
2Обработка документов 1 неделя ▶
POST /documents/upload — принять PDF, сохранить на диск
Извлечение текста через pymupdf или pdfplumber
Почему: качество извлечения влияет на всё что дальше
Разбивка на чанки: RecursiveCharacterTextSplitter из LangChain
Почему: правильный chunk size критичен для качества поиска (500–1000 символов)
Эмбеддинги через sentence-transformers (all-MiniLM-L6-v2)
Сохранить векторы в pgvector с метаданными (source, page, chunk_id)
💡 Тестируй каждый шаг отдельно через Python скрипт, не всё сразу. Извлечение → чанкинг → эмбеддинги → поиск.
3RAG-система и чат 1 неделя ▶
GET /search?q=... — семантический поиск по чанкам
Почему: убедиться что поиск работает до интеграции с LLM
Подключить Ollama через HTTP (OpenAI-совместимый API)
RAG пайплайн: вопрос → поиск топ-5 чанков → промпт → ответ LLM
Почему: это сердце всей системы
Базовый UI чата на Next.js, стриминг ответов через SSE
💡 Сначала проверь RAG через curl — потом делай UI. Стриминг даёт ощущение живого ответа.
4Конспекты и теги 3–4 дня ▶
После загрузки документа — автогенерация summary через LLM
Почему: пользователь сразу видит суть без чтения
Извлечение ключевых слов и тегов (промпт к LLM или KeyBERT)
Почему: теги нужны для фильтрации и рекомендаций в V2
Показывать конспект в UI при клике на документ
5Граф знаний 1 неделя ▶
NER — извлечение сущностей из текста (spacy или промпт к LLM)
Почему: без сущностей нет узлов в графе
LLM определяет связи: "X использует Y", "X объясняет Z", "X является частью Z"
Почему: связи делают граф полезным, не просто список слов
Граф в NetworkX, экспорт в JSON через /graph API
Визуализация в браузере через D3.js или vis.js
💡 Граф — это WOW-момент проекта. Именно он отличает тебя от обычного RAG и впечатляет рекрутеров.
6Долговременная память 1 неделя ▶
Логировать все запросы и просмотренные документы в БД
Аналитика паттернов: частые темы, давно не открытые материалы
Уведомления: "Ты не открывал RAG уже 20 дней"
Персональные рекомендации на основе истории
7Финализация и публикация 3–5 дней ▶
README с GIF-демо, архитектурной схемой, командой запуска
Почему: README — первое что видит рекрутер
GitHub Actions: линтер + тесты при каждом пуше
Видео-демо 2–3 минуты с реальным use-case
Метрики: время ответа, качество поиска, RAM/CPU
💡 Потрать на README столько же времени сколько на код. Это твоя витрина.
